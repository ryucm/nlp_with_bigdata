# BPE vs WordPiece 토크나이저 비교 평가

고려대학교 빅데이터자연어처리기술 과제 — 토크나이저 학습 및 성능 비교 리포트

> **라이브러리 vs 직접 구현 비교에 대하여**
> 이 노트북은 과제 요건 외에 라이브러리와 직접 구현 간의 성능 차이가 얼마나 나는지 개인적으로 궁금하여 추가로 비교한 실험을 포함하고 있습니다.
> 라이브러리의 각 코드 한 줄이 내부적으로 어떤 단계를 수행하는지, 직접 구현과 1:1로 대응시켜 확인했습니다.

---

## 과제 요건

| 항목 | 내용 |
|------|------|
| 토크나이저 | BPE, WordPiece, 한국어 형태소 분석기(Mecab, Okt, Kkma) |
| 학습 데이터 | 최소 3만 단어 또는 1MB 이상 |
| 평가 데이터 | 1~2KB |
| 평가 항목 1 | 학습 데이터 크기(10% / 50% / 100%)에 따른 성능 비교 |
| 평가 항목 2 | 전체 데이터 기준 속도 비교 |
| 제출 | 소스코드 + 분석 리포트 |

**daekeun-ml/naver-news-summarization-ko** (HuggingFace Datasets)

네이버 뉴스 기사 본문(`document` 컬럼)을 학습 데이터로 사용합니다.

| 학습 기사 수 | 22,173건 |
| 데이터 용량 | 약 21MB |

| 기사당 평균 단어 수 | 237개 |
| 총 단어 수 | 약 525만 개 |
| 평가 데이터 | 약 2KB (검증셋 앞부분) |

> 과제 요건인 "최소 3만 단어 또는 1MB 이상"을 크게 초과합니다.

## 실험 설계

### 공통 설정

| 설정 | 값 |
|------|----|
| VOCAB_SIZE | 8,000 |
| min_frequency | 2 |
| 특수 토큰 | `[UNK]`, `[PAD]`, `[CLS]`, `[SEP]`, `[MASK]` |

### 구현 방식

각 토크나이저를 두 가지 방식으로 구현하여 **라이브러리 내부 동작을 직접 확인**합니다.

| 방식 | 설명 |
|------|------|
| **라이브러리 구현** | HuggingFace `tokenizers` 라이브러리 사용 |
| **직접 구현** | 각 학습 단계를 Python으로 직접 구현 |

라이브러리 코드의 각 줄에는 직접 구현의 어느 단계에 해당하는지 주석으로 표시하고, 직접 구현의 각 단계에는 대응하는 라이브러리 코드를 함께 표시합니다.

## 평가 지표

| 지표 | 설명 | 좋은 값 |
|------|------|---------|
| **Fertility** | 단어당 평균 토큰 수 | 낮을수록 좋음 |
| **Compression Ratio** | 문자당 토큰 수 | 낮을수록 좋음 |
| **OOV Rate** | `[UNK]` 비율 | 낮을수록 좋음 |
| **Throughput** | 초당 처리 토큰 수 | 높을수록 빠름 |

---

## BPE vs WordPiece 알고리즘 비교

| 단계 | BPE | WordPiece |
|------|-----|-----------|
| 초기 분해 | `삼성전자` → `['삼','성','전','자']` | `삼성전자` → `['삼','##성','##전','##자']` |
| 병합 기준 | `count(AB)` 최대 (빈도) | `count(AB) / (count(A) × count(B))` 최대 (우도) |
| 토크나이징 | 학습된 병합 규칙을 순서대로 적용 | vocab에서 앞→뒤 방향으로 가장 긴 prefix 탐색 |
| 단어 내부 표시 | 없음 | `##` 접두사로 단어 내부 서브워드 명시 |

## 실험 결과

### 1. 데이터 크기별 성능 비교 (라이브러리 기준)

| 토크나이저 | 데이터 | Fertility | Compression Ratio | OOV Rate |
|-----------|--------|-----------|-------------------|----------|
| BPE | 10% | 1.7992 | 0.4377 | 0.00% |
| BPE | 50% | 1.7746 | 0.4317 | 0.00% |
| BPE | 100% | 1.7828 | 0.4337 | 0.00% |
| WordPiece | 10% | 1.9262 | 0.4686 | 0.00% |
| WordPiece | 50% | 1.9344 | 0.4706 | 0.00% |
| WordPiece | 100% | 1.9713 | 0.4796 | 0.00% |

- BPE는 데이터 크기와 무관하게 fertility가 안정적(1.80→1.78)입니다. 자주 나오는 쌍은 10%에서도 충분히 학습됩니다.
- WordPiece는 데이터가 늘수록 fertility가 소폭 증가(1.93→1.97)합니다. score 공식의 분모(개별 토큰 빈도)가 커져서 병합이 억제되기 때문입니다.

### 2. 라이브러리 vs 직접 구현 비교 (전체 데이터 기준)

| 토크나이저 | 구현 방식 | Fertility | Compression Ratio | OOV Rate | Throughput |
|-----------|----------|-----------|-------------------|----------|------------|
| BPE | 라이브러리 | 1.7828 | 0.4337 | 0.00% | 405,900 tok/s |
| BPE | 직접 구현 | 1.7172 | 0.4177 | 0.00% | 3,878 tok/s |
| WordPiece | 라이브러리 | 1.9713 | 0.4796 | 0.00% | 668,919 tok/s |
| WordPiece | 직접 구현 | 3.0246 | 0.7358 | 0.00% | 2,097,152 tok/s |

- BPE는 라이브러리와 직접 구현이 거의 동일한 결과를 냅니다. throughput 차이(약 105배)는 Rust vs Python 구현 언어의 차이입니다.
- WordPiece는 직접 구현의 fertility가 3.02로 라이브러리(1.97)보다 크게 높습니다. vocab 구성의 차이로 greedy longest match 결과가 달라졌기 때문입니다.

### 3. BPE vs WordPiece 종합 비교 (라이브러리, 전체 데이터)

| 지표 | BPE | WordPiece | 우위 |
|------|-----|-----------|------|
| Fertility | **1.7828** | 1.9713 | **BPE** |
| Compression Ratio | **0.4337** | 0.4796 | **BPE** |
| OOV Rate | 0.00% | 0.00% | 동일 |
| Throughput | 1,989,106 tok/s | **2,276,439 tok/s** | **WordPiece** |

BPE가 Fertility·Compression Ratio에서 우수한 이유: 뉴스 도메인에서 반복되는 고유명사(서울, 연합뉴스 등)를 빈도 기반으로 빠르게 단일 토큰으로 병합하기 때문입니다.

### 4. 속도 비교 (전체 데이터, 라이브러리 기준)

| 토크나이저 | 처리 토큰 수 | 소요 시간 | Throughput (tok/s) |
|-----------|------------|---------|-------------------|
| BPE | 9,175,688 | 4.61초 | 1,989,106 |
| WordPiece | 10,022,162 | 4.40초 | 2,276,439 |

WordPiece가 약 14.4% 빠릅니다. BPE는 병합 규칙을 순서대로 전부 적용해야 하고, WordPiece는 vocab을 한 번 조회하면 끝나는 greedy longest match 방식이기 때문입니다.

### 5. 형태소 분석기 비교 (Mecab / Okt / Kkma)

형태소 분석기는 별도 학습 없이 언어학 규칙 기반으로 동작합니다. 동일한 테스트셋에서 성능을 비교합니다.

| 분석기 | 방식 | Fertility | Compression Ratio | OOV Rate | Throughput |
|--------|------|-----------|-------------------|----------|------------|
| Mecab | C++ 규칙+통계 | 1.8361 | 0.4467 | N/A | 3,195 tok/s |
| Okt | Java (KoNLPy) | **1.6107** | **0.3918** | N/A | 480 tok/s |
| Kkma | Java (KoNLPy) | 2.1680 | 0.5274 | N/A | 334 tok/s |

- Fertility 기준: Okt(1.61) < Mecab(1.84) < Kkma(2.17)
- 속도 기준: Mecab(3,195) > Okt(480) > Kkma(334) tok/s
- Mecab은 C++ 기반으로 Java 기반 Okt·Kkma보다 약 7~10배 빠릅니다.
- 형태소 분석기는 고정된 사전을 사용하므로 OOV 개념이 BPE·WordPiece와 다릅니다.

## 실행 방법

```bash

cd nlp_with_bigdata
uv sync

# 2. Jupyter 실행
```
