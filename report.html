소스코드: https://github.com/ryucm/nlp_with_bigdata

라이브러리 vs 직접 구현 비교에 대하여
이 리포트는 과제 요건 외에, 라이브러리와 직접 구현 간의 성능 차이가 얼마나 나는지 개인적으로 궁금하여 추가로 비교한 실험을 포함하고 있습니다. 라이브러리의 각 코드 한 줄이 내부적으로 어떤 단계를 수행하는지, 직접 구현과 1:1로 대응시켜 확인했습니다.

4. BPE vs WordPiece 알고리즘 비교

단계BPEWordPiece
초기 분해 ['삼','성','전','자'] ['삼','##성','##전','##자']
병합 기준 count(AB) 최대 (빈도) count(AB) / (count(A) × count(B)) 최대 (우도)
토크나이징 학습된 병합 규칙을 순서대로 적용 vocab에서 앞→뒤 방향으로 가장 긴 prefix 탐색 (greedy longest match)
단어 내부 표시 없음 ## 접두사로 단어 내부 서브워드 명시

BPE가 Fertility·Compression Ratio에서 더 좋은 이유

BPE는 count(AB)가 가장 큰 쌍을 병합합니다. 뉴스 데이터에서 반복 등장하는 고유명사(서울, 연합뉴스, 현대백화점 등)가 빠르게 단일 토큰으로 합쳐집니다.

WordPiece는 count(AB) / (count(A) × count(B))를 기준으로 합니다. A, B가 각각 매우 자주 나오면 분모가 커져서 score가 낮아지고 병합이 억제됩니다. 결과적으로 더 작은 단위의 토큰이 vocab에 남아 하나의 단어를 더 많은 토큰으로 표현하게 됩니다.

샘플 문장: 서울 연합뉴스 현대백화점그룹이 광주광역시에 서울 여의도 더현대 서울 과 같은 문화복합몰을 만든다고 6일 밝혔다.

⑤BPE 라이브러리 10%–

② ['서울', '연합뉴스', '현대백화점', '그룹이', '광주', '광역시', '에', ...]

WordPiece가 BPE보다 약 14.4% 빠릅니다. BPE는 학습된 병합 규칙 목록을 처음부터 끝까지 순서대로 적용해야 하는 반면, WordPiece는 vocab에서 한 번 조회하면 끝나는 greedy longest match 방식이기 때문입니다. 두 알고리즘 모두 초당 약 200만 토큰 처리로 실용적인 속도 수준입니다.

6. 한국어 형태소 분석기 비교 (Mecab / Okt / Kkma)

형태소 분석기는 BPE·WordPiece와 달리 별도 학습 없이 언어학 규칙 기반으로 동작합니다. 동일한 테스트 데이터에 대해 세 분석기의 성능을 비교합니다.

분석기방식특징
MecabC++ 규칙+통계가장 빠름, 세밀한 형태소 분리
OktJava 기반 (KoNLPy)간단한 형태소 분리, 복합명사 처리
KkmaJava 기반 (KoNLPy)가장 정밀한 분석, 가장 느림

6.1 토크나이징 예시

샘플 문장: 서울 연합뉴스 현대백화점그룹이 광주광역시에 서울 여의도 더현대 서울 과 같은 문화복합몰을 만든다고 6일 밝혔다.

분석기결과 예시

Okt['서울', '연합뉴스', '현대', '백화점', '그룹', '이', '광주광역시', '에', ...]
Kkma['서울', '연합', '뉴스', '현대', '백화점', '그룹', '이', '광주', '광역시', '에', ...]

6.3 분석

Fertility 기준으로는 Okt(1.61) < Mecab(1.84) < Kkma(2.17) 순입니다.
Okt가 가장 낮은 이유는 복합명사를 통째로 하나의 토큰으로 처리하는 경우가 많기 때문입니다 (예: 광주광역시 → ['광주광역시']).
반면 Kkma는 언어학적으로 가장 세밀하게 분해하므로 fertility가 가장 높습니다 (예: 올랐다 → ['올르', '았', '다']).

속도(Throughput) 기준으로는 Mecab(3,195 tok/s) > Okt(480 tok/s) > Kkma(334 tok/s) 순입니다.
Mecab은 C++ 바이너리 기반이라 Java 기반인 Okt·Kkma보다 약 7~10배 빠릅니다.
Kkma는 세밀한 분석을 위해 더 많은 연산을 수행하므로 Okt보다도 느립니다.

실용적 선택 기준:
속도가 중요한 대용량 처리 환경에서는 Mecab이 적합합니다.
형태소 분석의 정밀도가 중요한 학술 연구에서는 Kkma가 적합합니다.
속도와 정확도의 균형이 필요하면 Okt가 무난합니다.

형태소 분석기는 고정된 사전과 언어학 규칙을 사용하므로 OOV 개념이 BPE·WordPiece와 다릅니다. 미등록 단어는 더 작은 단위로 분리되거나 그대로 출력됩니다.

7. 결론

BPE·WordPiece·형태소 분석기(Mecab, Okt, Kkma) 모두 한국어 뉴스 도메인에서 유효하게 동작하지만, 목적과 환경에 따라 적합한 방식이 다릅니다.

BPE vs WordPiece

성능 측면에서 BPE가 Fertility(1.78)와 Compression Ratio(0.43) 모두에서 WordPiece(1.97, 0.48)보다 우수합니다. 한국어 뉴스 도메인은 반복 등장하는 고유명사(지명, 기업명)가 많아, 순수 빈도 기반인 BPE가 이를 빠르게 단일 토큰으로 병합하기 때문입니다.

속도 측면에서는 WordPiece(2,276,439 tok/s)가 BPE(1,989,106 tok/s)보다 약 14% 빠릅니다.

형태소 분석기 (Mecab / Okt / Kkma)

세 분석기 중 Fertility 기준으로는 Okt(1.61) < Mecab(1.84) < Kkma(2.17) 순이며, 속도 기준으로는 Mecab(3,195 tok/s) > Okt(480 tok/s) > Kkma(334 tok/s) 순입니다.
Mecab은 C++ 기반으로 속도가 압도적이며, Okt는 복합명사를 통합 처리해 fertility가 가장 낮고, Kkma는 가장 정밀한 분석을 제공하지만 속도가 가장 느립니다.

라이브러리 vs 직접 구현에서, BPE는 두 방식이 거의 동일한 결과를 냅니다. WordPiece는 vocab 구성의 미묘한 차이로 직접 구현의 토크나이징 결과가 라이브러리와 크게 달라지며, 이는 라이브러리 내부에 추가적인 최적화 처리가 존재함을 보여줍니다.
